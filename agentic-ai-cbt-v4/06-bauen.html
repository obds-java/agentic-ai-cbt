Agenten bauen: Vorgehen & Frameworks

  • lesen, wie ein Tool und Memory in Spring AI 2.0 aussehen

6.1 Ein pragmatisches Vorgehen

Die wichtigste Regel kennst du schon: so einfach wie möglich anfangen. Daraus ergibt sich ein bewährter Pfad:

  1. Use Case schneiden: eng, klar messbar, echter Mehrwert. „Beantworte alle Kundenmails" ist zu breit; „erstelle aus einer Reklamationsmail einen strukturierten Ticket-Entwurf" ist machbar.
  2. Als Workflow starten: Verdrahte die Schritte, die du kennst. Du wirst überrascht sein, wie weit das trägt.
  3. Tools + Loop ergänzen, nur an den Stellen, wo die Varianz echte Autonomie verlangt.
  4. Von Anfang an tracen (Modul 8) – nicht erst, wenn es brennt.
  5. Härten: Guardrails, Oversight (Modul 10), Permissions & Security (Modul 11), bevor es echte Nutzer oder echte Daten berührt.

6.3 Wie das in Spring AI aussieht

Version beachten – APIs sind in Bewegung

Die folgenden Fragmente sind für Spring AI 2.0.0 (2026) geschrieben. Wichtig: Der frühere ToolCallAdvisor wurde in ToolCallingAdvisor umbenannt (der alte Name existiert nur noch deprecated). In 2.0 wird die Tool-Ausführung über den ChatClient mit ToolCallingAdvisor abgewickelt – ein Default wird automatisch verdrahtet, sofern nicht explizit deaktiviert. Die Fragmente folgen der 2.0-Dokumentation, wurden aber nicht in einem Projekt kompiliert: vor dem Übernehmen immer gegen die eigene Version bauen – sonst ist Code „fast richtig, aber nicht ausführbar".

Ein Tool ist einfach eine annotierte Methode

Spring AI generiert das JSON-Schema aus deiner Methode und übernimmt das Parameter-Mapping über die @Tool-Annotation:

OrderService.javaSpring AI 2.0.0
@Service
public class OrderService {

    @Tool(description = "Liefert den aktuellen Versandstatus einer Bestellung per ID.")
    public String getOrderStatus(String orderId) {
        return orderRepository.findById(orderId)
            .map(Order::getShippingStatus)
            .orElseThrow(() -> new OrderNotFoundException(orderId));
    }
}

Die description ist kein Kommentar fürs Team, sondern das, woran das LLM erkennt, wann es dieses Tool nutzen soll (siehe Modul 4). Gute Beschreibung = gute Tool-Nutzung.

Memory & Tools über Advisors zusammenstecken

Spring AI nutzt das Advisor-Muster: Memory, Tool-Calling und mehr sind komponierbare Bausteine am ChatClient – das Harness in Spring-Sprache.

AgentConfig.javaSpring AI 2.0.0
ChatClient chatClient = chatClientBuilder
    .defaultAdvisors(
        // Kurzzeitgedächtnis: gleitendes Fenster über die Konversation
        MessageChatMemoryAdvisor.builder(
            MessageWindowChatMemory.builder().maxMessages(100).build())
            .build())
    // Tool-Calling-Loop: in 2.0 auto-registriert (ToolCallingAdvisor),
    // Tools einfach als Beans/Objekte übergeben:
    .defaultTools(new OrderService())
    .build();
Der Aha-Moment für Java-Devs

Ein Agent in Spring AI ist kein exotisches Paradigma – es ist Dependency Injection + ein Loop + gut beschriebene Methoden. Das meiste kannst du schon. Neu ist vor allem das Denken in Kontext, Tools und Nicht-Determinismus.

6.4 Context Engineering — die zentrale Stellschraube

Der erste Reflex vieler Teams: alles ins Kontextfenster kippen, „sicherheitshalber". Das ist fast immer ein Fehler. Über Kosten und Latenz hinaus sinkt die Fähigkeit des Modells, einzelne Informationen zu beachten, je größer der Input wird. Teams haben erlebt, dass ein Agent mit einer 100K-Token-Zusammenfassung schlechter abschnitt als mit gezielten 5K Token zur selben Aufgabe.

Merksatz
Context Engineering heißt nicht „möglichst viel reingeben", sondern „das Richtige reingeben". Der Kontext ist ein knappes Budget – kuratiere ihn wie ein Redakteur, nicht wie ein Messie.
