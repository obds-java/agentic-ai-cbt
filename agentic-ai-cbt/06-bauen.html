Agenten bauen: Vorgehen & Frameworks

Vom Problem zum laufenden Agenten – mit einem pragmatischen Vorgehen, einem Überblick über die Framework-Landschaft und kurzen Spring-AI-Fragmenten für unseren Stack.

Nach diesem Modul kannst du …

  • ein bewährtes, schrittweises Vorgehen anwenden (workflow-first)

  • lesen, wie ein Tool und Memory in Spring AI aussehen
  • Context Engineering als zentrale Stellschraube erkennen

6.1 Ein pragmatisches Vorgehen

Die wichtigste Regel kennst du schon aus Modul 1: so einfach wie möglich anfangen. Daraus ergibt sich ein bewährter Pfad:

  1. Use Case schneiden: eng, klar messbar, echter Mehrwert. „Beantworte alle Kundenmails" ist zu breit; „erstelle aus einer Reklamationsmail einen strukturierten Ticket-Entwurf" ist machbar.
  2. Als Workflow starten: Verdrahte die Schritte, die du kennst. Du wirst überrascht sein, wie weit das trägt.
  3. Tools + Loop ergänzen, nur an den Stellen, wo die Varianz echte Autonomie verlangt.
  4. Von Anfang an tracen (Modul 5) – nicht erst, wenn es brennt.
  5. Härten: Guardrails, Berechtigungen, Security (Modul 9), bevor es echte Nutzer oder echte Daten berührt.

Merksatz aus der Praxis

„Don't use a hammer to turn a screw." n8n für Ops & App-Verkettung, Flowise für RAG-Bots, LangFlow für visuelles Prototyping von Logik. Für Produktions-Backends in unserem Haus ist Spring AI der natürliche Andockpunkt.

6.3 Wie das in Spring AI aussieht

Du musst das hier nicht zum Laufen bringen – es geht ums Lesen und Wiedererkennen. Auffällig: Wie wenig „KI-Magie" und wie viel vertrautes Spring drinsteckt.

Ein Tool ist einfach eine annotierte Methode

Spring AI generiert das JSON-Schema aus deiner Methode und übernimmt das Parameter-Mapping über die @Tool-Annotation. Mehr braucht es nicht, damit der Agent diese Funktion aufrufen kann:

OrderService.java
@Service
public class OrderService {

    @Tool(description = "Liefert den aktuellen Versandstatus einer Bestellung per ID.")
    public String getOrderStatus(String orderId) {
        return orderRepository.findById(orderId)
            .map(Order::getShippingStatus)
            .orElseThrow(() -> new OrderNotFoundException(orderId));
    }
}

Die description ist hier kein Kommentar fürs Team, sondern das, woran das LLM erkennt, wann es dieses Tool nutzen soll (siehe Modul 4). Gute Beschreibung = gute Tool-Nutzung.

Memory & Tools über Advisors zusammenstecken

Spring AI nutzt das Advisor-Muster: Memory, Tool-Calling und mehr sind komponierbare Bausteine, die du in den ChatClient einhängst – das Harness in Spring-Sprache.

AgentConfig.java
ChatClient chatClient = chatClientBuilder
    .defaultAdvisors(
        // Kurzzeitgedächtnis: gleitendes Fenster über die Konversation
        MessageChatMemoryAdvisor.builder(
            MessageWindowChatMemory.builder().maxMessages(100).build())
            .build(),
        // Tool-Calling-Loop
        ToolCallAdvisor.builder().build())
    .defaultTools(new OrderService())
    .build();
Der Aha-Moment für Java-Devs

Ein Agent in Spring AI ist kein exotisches neues Paradigma – es ist Dependency Injection + ein Loop + gut beschriebene Methoden. Das meiste, was du brauchst, kannst du schon. Neu ist vor allem das Denken in Kontext, Tools und Nicht-Determinismus.

6.4 Context Engineering — die zentrale Stellschraube

Der erste Reflex vieler Teams: alles ins Kontextfenster kippen, „sicherheitshalber". Das ist fast immer ein Fehler. Über die Kosten- und Latenznachteile hinaus sinkt die Fähigkeit des Modells, einzelne Informationen zu beachten, je größer der Input wird. Teams haben erlebt, dass ein Agent mit einer 100K-Token-Zusammenfassung schlechter abschnitt als mit gezielten 5K Token zur selben Aufgabe.

Merksatz
Context Engineering heißt nicht „möglichst viel reingeben", sondern „das Richtige reingeben". Der Kontext ist ein knappes Budget – kuratiere ihn wie ein Redakteur, nicht wie ein Messie.
