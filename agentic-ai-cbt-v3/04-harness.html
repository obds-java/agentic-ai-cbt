4 Kernthema

Das Harness im Detail

Hier entscheidet sich, ob ein Agent ein Demo-Spielzeug bleibt oder produktionsreif wird. Memory, Tools, Skills, MCP, Observability, Guardrails – und warum das Harness zum Wettbewerbshebel geworden ist.

Nach diesem Modul kannst du …

  • die Hauptbausteine eines Harness benennen und ihre Aufgabe erklären
  • verstehen, was MCP ist und welches Problem es löst – und was RAG bedeutet
  • den Unterschied zwischen Tools, Skills und Memory sauber ziehen
  • begründen, warum „Harness Engineering" an Bedeutung gewinnt

4.1 Was ist das Harness?

Das Harness ist die komplette Software-Infrastruktur, die ein LLM umschließt: der Orchestrierungs-Loop, die Tools, das Gedächtnis, das Kontext-Management, die Zustands-Persistenz, die Fehlerbehandlung und die Guardrails. Kurz: Das Harness ist alles, was aus einem rohen Modell ein verlässlich arbeitendes System macht.

Zur Einordnung des Begriffs

Der Begriff „Harness" wird 2025/2026 im Umfeld von Coding Agents und produktiven Agentensystemen deutlich häufiger verwendet – besonders bei Anthropic. Das Konzept selbst ist älter; neu ist vor allem, dass es einen eigenen Namen und bewusste Engineering-Praxis bekommen hat. „Harness" ist also kein über Nacht etabliertes Standardvokabular, aber ein klar im Aufwind befindlicher, nützlicher Begriff.

4.2 Die Bausteine

🧠 Memory — Gedächtnis

Das Modell ist zustandslos. Memory ist das, was Kontinuität schafft. Man unterscheidet grob: Kurzzeitgedächtnis (der Gesprächsverlauf im aktuellen Kontextfenster) und Langzeitgedächtnis (Fakten, die über Sessions hinweg überleben – z. B. in Dateien, einer DB oder einem Vektorspeicher). Die Königsdisziplin: Was kommt ins Kontextfenster und was bleibt draußen? Das ist Context Engineering (Module 6 & 9).

Exkurs: Was ist eigentlich RAG?

RAG = Retrieval-Augmented Generation: Bevor (oder während) das Modell antwortet, sucht das System die relevanten Ausschnitte aus einer Wissensbasis – oft per semantischer Suche in einer Vektor-Datenbank – und legt sie dem Modell in den Kontext. Das Modell „weiss" die Firmenpolicy also nicht, es bekommt sie hingelegt. Bei Agenten ist Retrieval typischerweise einfach ein weiteres Tool („suche in der Wissensbasis"), das der Agent bei Bedarf selbst aufruft. Wann immer in dieser CBT „RAG für Policies" o. Ä. steht, ist genau das gemeint.

🔧 Tools — Werkzeuge

Tools sind die Hände des Agenten. Sie werden als Schemata definiert (Name, Beschreibung, Parametertypen) und in den Kontext des LLM injiziert, damit das Modell weiß, was verfügbar ist. Die Beschreibung eines Tools ist überraschend wichtig: Ein schlecht beschriebenes Tool wird falsch oder gar nicht genutzt. Tool-Design ist halb Doku-Schreiben, halb API-Design. Und nicht vergessen: Tools werden irgendwo ausgeführt – in welcher Umgebung (Sandbox, Container) und mit welchen Rechten, ist Teil des Harness-Designs (mehr in Modul 11).

📦 Skills — Fertigkeiten

Skills sind verpackte, wiederverwendbare Anleitungen oder Prozeduren – „so erstellst du einen Quartalsbericht", inklusive nötiger Schritte, Vorlagen und ggf. zugehöriger Tools. Wo ein Tool eine einzelne Funktion ist, bündelt ein Skill Wissen über ein Vorgehen.

🔌 MCP — Model Context Protocol

Damit Tools verlässlich und wiederverwendbar werden, definiert Anthropics Model Context Protocol (MCP) eine standardisierte, JSON-RPC-basierte Schnittstelle zwischen Modellen und externen Ressourcen. Die Analogie: MCP ist der „USB-C-Anschluss" für KI – statt für jede Kombination aus Agent und System einen eigenen Adapter zu bauen, sprechen beide Seiten dasselbe Protokoll. Ein MCP-Server für Jira, Slack oder eine interne DB lässt sich so an beliebige MCP-fähige Agenten anstecken. Wichtig: MCP selbst betont User Consent, Data Privacy und Tool Safety – Tools sind potenziell beliebige Codeausführung (mehr dazu in Modul 11).

Tool vs. Skill vs. MCP – die saubere Abgrenzung

Tool = eine einzelne aufrufbare Funktion. Skill = verpacktes Vorgehenswissen, das mehrere Tools und Schritte bündelt. MCP = das Protokoll, über das Tools/Ressourcen standardisiert angebunden werden. MCP ist also kein Werkzeug, sondern die Steckdosennorm.

📊 Observability — Sicht ins Innere

Observability ist die durchgängige Sichtbarkeit jedes Schritts, den ein Agent macht: LLM-Calls, Tool-Aufrufe, Retrieval, Planungsentscheidungen. Ohne sie fliegt ein produktiver Agent blind. Eigenes Modul (Modul 8) – inklusive Evals.

🛡️ Guardrails — Leitplanken

Guardrails sind die Grenzen: Welche Aktionen darf der Agent ohne Rückfrage ausführen? Welche Eingaben/Ausgaben werden gefiltert? Wann braucht es ein menschliches „OK" (Human-in-the-Loop)? Guardrails sind teils Policy, teils Code – und eng mit Oversight (Modul 10) und Security (Modul 11) verzahnt.

🧭 System-Prompt — die Instruktionsebene

Oft übersehen, aber ein echter Harness-Baustein: der System-Prompt. Er definiert Rolle, Arbeitsweise, Regeln und Grenzen des Agenten („Du bist ein Support-Agent. Nutze für Rechnungen immer Tool X. Sende nie ohne Freigabe.") – gesetzt vom Entwickler, unsichtbar für den Endnutzer. In den Hands-On Labs (Modul 7) wirst du erleben, wie stark eine einzige Zeile System-Prompt das Verhalten verändert. Er ist die weiche Verhaltenssteuerung; harte Grenzen gehören zusätzlich in Guardrails und Permissions – Prompt-Anweisungen allein sind kein Sicherheitsmechanismus (Modul 11).

🧯 Fehlerbehandlung — der stille Querschnitt

Tools schlagen fehl, APIs sind langsam, Antworten sind mal unbrauchbar. Ein gutes Harness behandelt das systematisch: Fehler als Beobachtung zurückspielen (damit der Agent umplanen kann, statt blind weiterzumachen), Retries mit Limit für flüchtige Fehler, idempotente Tools (damit ein Retry nichts doppelt ausführt – wichtig ab Modul 10) und klare Abbruchpfade, wenn nichts mehr geht. Der häufigste Anfängerfehler ist das Gegenteil: Exceptions verschlucken und dem Modell eine heile Welt vorspielen.

4.3 Warum das Harness zum Hebel wird

Drei Entwicklungen treffen zusammen:

  • Modelle werden zunehmend austauschbarer. Der Reasoning-Sprung der letzten Generationen bedeutet: Für viele Aufgaben ist das Gehirn nicht mehr der Engpass. Der Differenzierungshebel verschiebt sich Richtung Harness, Tooling und Betrieb – ohne dass die Modellwahl irrelevant würde.
  • Harness Engineering wird zur Disziplin. Es gibt etablierte Muster, Bibliotheken und kuratierte Listen rund um Harness-Engineering: Tools, Permissions, Memory, MCP, Observability, Orchestrierung.
  • Standardisierung greift. MCP für Tool-Anbindung und OpenTelemetry für Telemetrie schaffen Interoperabilität.
Merksatz
Modelle werden austauschbarer, aber nicht irrelevant. Der Wettbewerbshebel verschiebt sich Richtung Harness, Tooling, Evals und Betrieb – also genau dorthin, wo dein Team Einfluss hat.
