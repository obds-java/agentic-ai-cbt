7 Hands-On

Hands-On Labs

Genug Theorie. Lab 1 ist Pflicht und geführt – ein einziger, möglichst robuster und getesteter Pfad. Lab 2 ist optional für den Transfer in unseren Java/Spring-Stack.

Warum nur ein Pfad?

Statt mehrerer Wege, die nur Varianz und Frust erzeugen, gibt es genau einen getesteten Pfad: genau ein Tool (Flowise, fixiert auf Version 3.1.3), genau ein Werkzeug (Calculator), keine Websuche (die bräuchte Zusatz-Keys). Der Pfad wurde von einem Kollegen aus dem Team real durchgespielt. So bist du in ~20 Minuten durch und siehst den Agent-Loop live – und falls doch etwas klemmt, hilft die Troubleshooting-Box unten. (UI-Beschriftungen und Limits: Stand Juli 2026.)

Lab 1 Flowise · geführter Agent mit Calculator Pflicht

Ziel: Du baust einen Chat-Agenten, der selbstständig ein Rechen-Tool aufruft, und siehst im Log die Abfolge denken → Tool rufen → Ergebnis → antworten. Dauer ~20 Min, kostenlos.

Schritt 0 · Flowise starten

Empfohlener Weg: Docker mit fixierter Version. Damit umgehst du sämtliche Node-Versionsprobleme und hast garantiert dieselbe Flowise-Version wie diese Anleitung:

TerminalFlowise 3.1.3
docker run -d \
  --name flowise \
  -p 3000:3000 \
  -v "$HOME/.flowise:/root/.flowise" \
  flowiseai/flowise:3.1.3

Danach http://localhost:3000 im Browser öffnen (der erste Start kann eine Minute dauern). Das Volume sorgt dafür, dass deine Flows einen Container-Neustart überleben. Kein Docker auf dem Gerät? Dann nimm den kostenlosen Cloud-Tier auf flowiseai.com (Login nötig) – gleiche Übung, keine Installation.

Bitte nicht: npx-Roulette

Von npx flowise start raten wir für dieses Lab ab: Flowise 3.1.x bringt eine native Abhängigkeit (better-sqlite3) mit, die je nach Node-Version (z. B. Node 24) den Start verweigert – das kostet genau die Tüftelzeit, die dieses Lab vermeiden will. Docker oder Cloud, fertig.

Erwartetes Ergebnis: Du siehst die Flowise-Oberfläche; in der Seitenleiste gibt es u. a. „Agentflows" und „Chatflows" (beide Listen noch leer).

Schritt 1 · Modell-Zugang (Credential) hinterlegen

Du brauchst Zugang zu einem Chat-Modell. Am einfachsten und kostenlos: GitHub Models – ihr habt den GitHub-Account ja ohnehin, keine Kreditkarte nötig (Empfehlung unten). Alternativ ein API-Key mit Gratis-/Restguthaben bei OpenAI, Anthropic oder Google. → Das ist Schicht 1 aus Modul 3: das Gehirn.

Vorab: nur Lab-Daten & Wegwerf-Token

Bevor du gleich ein Token erzeugst: Verwende ein persönliches Wegwerf-/Lab-Token und ausschliesslich Testdaten – keine Firmen- oder Kundendaten, keine produktiven API-Keys. Ein Lab-Token reicht vollständig. (Warum das wichtig ist, klärt Modul 11.)

Empfohlen: kostenlos mit GitHub Models

GitHub Models ist OpenAI-kompatibel – man hängt es in Flowise einfach an den ChatOpenAI-Knoten und überschreibt die Basis-URL. So geht's:

1. Token erzeugen: GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens → neuen Token mit Scope models:read erstellen.
2. In Flowise: als Chat-Modell ChatOpenAI wählen (nicht Anthropic/Google) – in Flowise 3.x direkt im Agent-Knoten (Schritt 3). In den Credentials den Token als „OpenAI API Key" eintragen.
3. Basis-URL überschreiben: im Knoten das Feld Base Path / BasePath auf https://models.github.ai/inference setzen. Falls du das Feld nicht siehst: ChatOpenAI-Knoten öffnen, Additional Parameters aufklappen und dort nach Base Path / Base URL suchen (Name variiert je nach Flowise-Version).
4. Modell wählen: bevorzugt openai/gpt-4o-mini bzw. gpt-4o-mini (Schreibweise je nach Flowise-Version). Wichtig: Nimm ein Modell, das Tool-/Function-Calling unterstützt – nicht jedes Modell im Katalog kann das gleich zuverlässig, und ohne Tool-Support scheitert Schritt 5. Exakte Modell-IDs und Verfügbarkeit im aktuellen GitHub-Models-Katalog prüfen.

Zu den Limits – wichtig (Stand Juli 2026): Bei GitHub Models sind nur ~50 API-Calls pro Tag kostenfrei (Richtwert für GPT-4o; variiert je nach Modell und GitHub-Plan). Das gilt pro Person – jeder nutzt sein eigenes Token, es ist kein geteilter Topf. Für dieses Lab reicht das locker: Ein Durchlauf braucht nur ~2–4 Calls, also ~15–25 Testläufe pro Tag. Tipp: gpt-4o-mini wählen – meist höheres Limit und für den Calculator-Agenten mehr als schlau genug. Für echte Agenten mit langen Loops wären 50/Tag zu wenig; dafür gibt es kostenpflichtige Nutzung mit Budget. Token wie ein Passwort behandeln – nicht ins Repo committen.

Alternative: dein eigenes Modell über dessen API einbinden

Du hast bereits einen API-Key bei OpenAI, Anthropic oder Google? Dann bind dein Modell direkt ein – Flowise hat dafür fertige Knoten: ChatOpenAI, ChatAnthropic, ChatGoogleGenerativeAI. Passenden Knoten wählen, deinen Key als Credential eintragen, Modell setzen (z. B. gpt-4o-mini, ein Claude-haiku, ein Gemini-flash) – fertig. Eine Basis-URL musst du dann nicht überschreiben.

Achtung – Abo ≠ API-Key: Ein Chat-Abo (ChatGPT Plus, Claude Pro, Gemini Advanced) enthält keinen API-Zugang. Den API-Key holst du dir separat im Entwickler-Portal des Anbieters (platform.openai.com, console.anthropic.com bzw. Google AI Studio); dort wird pro Nutzung abgerechnet – meist nur Cent-Beträge. Gratis-Tipp: Google AI Studio bietet für die Gemini-API einen echten kostenlosen Tier mit großzügigeren Limits als GitHub Models – eine gute Alternative, wenn dir 50/Tag zu knapp sind.

Erwartetes Ergebnis: Die Credential erscheint gespeichert in der Liste; sie lässt sich später im Knoten auswählen.

Schritt 2 · Einen Agentflow anlegen

In der Seitenleiste Agentflows öffnen → Add New. Du bekommst einen Canvas mit einem Start-Knoten. Füge einen Agent-Knoten hinzu (über das +/Knoten-Menü) – er wird automatisch mit Start verbunden. → Das ist Schicht 2: der Loop.

Erwartetes Ergebnis: Im Canvas stehen Start → Agent. Mehr Knoten brauchst du nicht.

Schritt 3 · Den Agent-Knoten konfigurieren: Modell + Calculator

Wichtig zu wissen: In Flowise 3.x (Agentflow V2) wird alles im Agent-Knoten selbst konfiguriert – Modell, Tools und System-Prompt. Es gibt keine separaten Knoten zu verbinden. Klick also auf den Agent-Knoten und stelle ein:

  • Model: ChatOpenAI wählen, deine Credential aus Schritt 1 zuweisen; bei GitHub Models zusätzlich Base Path und Modell-ID setzen (siehe Kasten in Schritt 1).
  • Tools: auf Add Tool (bzw. das Tools-Feld) klicken und Calculator hinzufügen. Sonst nichts – keine Websuche, keine Vektor-DB. → Ein Tool aus dem Harness (Modul 4), bewusst minimal.

Erwartetes Ergebnis: Im Agent-Knoten sind ein Modell (mit Credential) und das Tool „Calculator" eingetragen.

Deine Flowise-UI sieht anders aus?

Ältere Flowise-Versionen (bzw. der „Chatflows"-Bereich) arbeiten mit separaten Knoten, die man verbindet: ein Tool Agent-Knoten, daran ein ChatOpenAI-Knoten als Chat Model und ein Calculator-Knoten am Tool-Eingang. Das Ergebnis ist dasselbe. Wenn deine Oberfläche nicht zur Anleitung passt, prüfe zuerst die Version – mit dem Docker-Befehl aus Schritt 0 (3.1.3) stimmt sie garantiert überein.

Schritt 4 · System-Prompt setzen

Ebenfalls im Agent-Knoten: Trage im Feld Instructions / System Prompt ein kurzes Ziel ein, z. B.: Du bist ein hilfreicher Assistent. Nutze das Calculator-Tool für jede Rechnung, statt selbst zu rechnen. Flow speichern. → Die Instruktionsebene aus Modul 4 – du wirst gleich sehen, wie stark sie wirkt.

Schritt 5 · Testen – das Tool erzwingen

Chat öffnen und fragen: Was ist 1348 × 27 + 99?

Erwartetes Ergebnis: Die Antwort lautet 36 495. Wichtiger als die Zahl: im Verlauf/Log siehst du, dass der Agent das Calculator-Tool aufgerufen hat (mit dem Ausdruck als Parameter) und erst danach geantwortet hat.

Stelle danach noch eine zweite Frage, die ein Modell „im Kopf" kaum sicher löst: Rechne 17.5 % von 2840 und addiere 123.45.

Erwartetes Ergebnis: 620.45 (497 + 123.45) – wieder mit sichtbarem Calculator-Aufruf. Genau dafür sind Tools da: exakte Arithmetik ist keine Stärke von Sprachmodellen, und hier merkt man den Unterschied deutlicher als bei einer glatten Multiplikation.

Schritt 6 · Den Loop sichtbar machen

Öffne die Detailansicht des letzten Laufs (in Flowise meist „View Messages" / das Log-/Trace-Panel). Du erkennst die Sequenz:

run-log (vereinfacht)
# Reason  Agent: „Das ist eine Rechnung -> ich nutze Calculator."
# Act     tool: Calculator("1348 * 27 + 99")
# Observe result: 36495
# Final   Agent: „1348 × 27 + 99 = 36495."

Das ist der Reason-Act-Observe-Loop aus Modul 2 – live, an deinem eigenen Agenten.

Troubleshooting

„Invalid API key" / 401: Credential aus Schritt 1 nochmals prüfen; richtigen Modell-Typ (ChatOpenAI) gewählt? Key ohne Leerzeichen eingefügt?

Agent antwortet, ohne das Tool zu nutzen: System-Prompt schärfen („nutze immer das Calculator-Tool für Rechnungen"); prüfen, ob Calculator im Agent-Knoten unter Tools wirklich eingetragen ist. Bei GitHub Models: unterstützt dein gewähltes Modell Tool-Calling? Im Zweifel openai/gpt-4o-mini nehmen.

Die UI passt nicht zur Anleitung: Du hast vermutlich eine andere Flowise-Version (oder bist im „Chatflows"-Bereich statt „Agentflows"). Mit dem Docker-Befehl aus Schritt 0 bekommst du exakt Version 3.1.3 – oder nutze den Kasten „Deine Flowise-UI sieht anders aus?" in Schritt 3.

Docker startet nicht / kein Docker erlaubt: Nicht tüfteln → Cloud-Tier auf flowiseai.com nehmen.

npx flowise start bricht ab (z. B. Fehler um better-sqlite3): Bekanntes Problem mit neueren Node-Versionen (z. B. Node 24). Nicht debuggen – Docker-Befehl aus Schritt 0 verwenden.

Port 3000 belegt: im Docker-Befehl -p 3100:3000 verwenden und http://localhost:3100 öffnen.

Kein API-Key verfügbar: Viele Anbieter geben Gratis-Startguthaben; sonst Kollegen fragen oder den Cloud-Tier mit dortigem Free-Kontingent nutzen.

Abschlussaufgabe (das eigentliche Lernen)

Experiment: Tool-Beschreibung verschlechtern

Ändere die Beschreibung/den Namen des Calculator-Tools in etwas Nichtssagendes (z. B. „tool_x", Beschreibung leeren). Stelle dieselbe Rechenfrage erneut. Beobachtung: Der Agent nutzt das Tool nun seltener, falsch – oder rechnet selbst (und liegt evtl. daneben). Damit hast du am eigenen Agenten erlebt, warum Tool-Beschreibungen Teil des Harness-Engineerings sind (Modul 4).

Lab 1 geschafft, wenn …

  • dein Agent das Calculator-Tool von selbst aufgerufen hat
  • du im Log die Sequenz denken → Tool → Ergebnis → Antwort erkannt hast
  • du gesehen hast, wie eine schlechtere Tool-Beschreibung das Verhalten verändert

Lab 2 Spring-AI-Mini-Agent optional · Java-Transfer

Für die, die den Bezug zu unserem Stack wollen. Ziel: ein winziger Spring-Boot-Agent mit einem Tool (getOrderStatus) und Kurzzeitgedächtnis – lokal startbar, eine Frage, ein sichtbarer Tool-Call im Log.

Voraussetzungen & Version

JDK 17+, Maven, ein LLM-API-Key als Umgebungsvariable. Geschrieben für Spring AI 2.0.0 auf Spring Boot 4.0.x – Spring AI 2.0.x setzt Boot 4.0/4.1 (Framework 7) voraus. Läuft euer Projekt noch auf Boot 3.4/3.5? Dann gehört ihr zur Spring-AI-1.x-Linie: spring-ai-bom 1.1.x verwenden – die hier gezeigten APIs (@Tool, ChatClient, Memory-Advisor) sind dort weitgehend identisch. Bei Abweichungen die Reference-Docs eurer Version konsultieren.

Exakte URL/Path/Modell-ID bitte gegen die aktuelle GitHub-Models-Doku und deine Spring-AI-Version prüfen.

3 · Tool + Agent + eine Frage

MiniAgent.javaSpring AI 2.0.0
@SpringBootApplication
public class MiniAgent {
  public static void main(String[] a){ SpringApplication.run(MiniAgent.class, a); }

  // --- das eine Tool ---
  static class OrderTools {
    @Tool(description = "Liefert den Versandstatus einer Bestellung per ID.")
    String getOrderStatus(String orderId) {
      return "4711".equals(orderId) ? "versandt am 24.06." : "unbekannt";
    }
  }

  // --- Agent: ChatClient mit Memory; Tools werden in 2.0 automatisch geloopt ---
  @Bean
  CommandLineRunner run(ChatClient.Builder builder) {
    ChatClient agent = builder
        .defaultAdvisors(MessageChatMemoryAdvisor
            .builder(MessageWindowChatMemory.builder().build()).build())
        .defaultTools(new OrderTools())
        .build();

    return args -> {
      String answer = agent.prompt()
          .user("Wie ist der Status von Bestellung 4711?")
          .call().content();
      System.out.println(answer);
    };
  }
}

4 · Starten & den Tool-Call im Log sehen

Terminal
$ export OPENAI_API_KEY=sk-...
$ ./mvnw spring-boot:run
...
DEBUG o.s.a.tool  : calling tool 'getOrderStatus' args={orderId=4711}
DEBUG o.s.a.tool  : tool result: versandt am 24.06.
> Die Bestellung 4711 wurde am 24.06. versandt.

Tipp: logging.level.org.springframework.ai=DEBUG setzen, um Tool-Calls im Log zu sehen – euer erster, einfachster „Trace" (mehr in Modul 8).

Kleine Transferaufgabe

Verbessere die @Tool-Beschreibung (z. B. präziser, mit Hinweis aufs ID-Format) und stelle eine knifflige Frage („Status der Bestellung mit Nummer 4711?"). Beobachte im DEBUG-Log, ob das Tool zuverlässiger getroffen wird. Gleiche Lektion wie in Lab 1 – diesmal in Java.

Merksatz
Der Moment, in dem du im Log siehst, wie der Agent selbst entscheidet ein Tool zu rufen, ist der Moment, in dem „Agentic AI" vom Buzzword zur greifbaren Sache wird – ob im No-Code-Canvas oder in Spring.
