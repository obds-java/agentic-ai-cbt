7 Hands-On

Hands-On Labs

Genug Theorie. Lab 1 ist Pflicht und geführt – ein einziger, getesteter Pfad mit Erfolgsgarantie. Lab 2 ist optional für den Transfer in unseren Java/Spring-Stack.

Warum nur ein Pfad?

Statt mehrerer Wege, die nur Varianz und Frust erzeugen, gibt es genau einen getesteten Pfad: genau ein Tool (Flowise), genau ein Werkzeug (Calculator), keine Websuche (die bräuchte Zusatz-Keys). So bist du in ~20 Minuten durch und siehst den Agent-Loop garantiert live.

Lab 1 Flowise · geführter Agent mit Calculator Pflicht

Ziel: Du baust einen Chat-Agenten, der selbstständig ein Rechen-Tool aufruft, und siehst im Log die Abfolge denken → Tool rufen → Ergebnis → antworten. Dauer ~20 Min, kostenlos.

Schritt 0 · Flowise starten

Du hast zwei gleichwertige Optionen – nimm die, die auf deinem Gerät erlaubt ist:

  • Lokal (empfohlen, wenn Node ≥ 18 vorhanden): Terminal öffnen, npx flowise start ausführen, dann http://localhost:3000 im Browser öffnen.
  • Cloud: bei flowiseai.com den kostenlosen Cloud-Tier nutzen (Login nötig).

Erwartetes Ergebnis: Du siehst die Flowise-Oberfläche mit einer leeren Liste von „Chatflows"/„Agentflows".

Schritt 1 · Modell-Zugang (Credential) hinterlegen

Du brauchst Zugang zu einem Chat-Modell. Am einfachsten und kostenlos: GitHub Models – ihr habt den GitHub-Account ja ohnehin, keine Kreditkarte nötig (Empfehlung unten). Alternativ ein API-Key mit Gratis-/Restguthaben bei OpenAI, Anthropic oder Google. → Das ist Schicht 1 aus Modul 3: das Gehirn.

Empfohlen: kostenlos mit GitHub Models

GitHub Models ist OpenAI-kompatibel – man hängt es in Flowise einfach an den ChatOpenAI-Knoten und überschreibt die Basis-URL. So geht's:

1. Token erzeugen: GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens → neuen Token mit Scope models:read erstellen.
2. In Flowise: als Modell-Knoten ChatOpenAI wählen (nicht den Anthropic/Google-Knoten). In den Credentials den Token als „OpenAI API Key" eintragen.
3. Basis-URL überschreiben: im Knoten das Feld Base Path / BasePath (manchmal unter „Additional Parameters") auf https://models.github.ai/inference setzen.
4. Modell wählen: z. B. gpt-4o-mini oder openai/gpt-4o-mini (Schreibweise je nach Flowise-Version; im GitHub-Models-Katalog steht die exakte Modell-ID).

Zu den Limits – wichtig: Bei GitHub Models sind nur ~50 API-Calls pro Tag kostenfrei (Richtwert für GPT-4o; variiert je nach Modell und GitHub-Plan). Das gilt pro Person – jeder nutzt sein eigenes Token, es ist kein geteilter Topf. Für dieses Lab reicht das locker: Ein Durchlauf braucht nur ~2–4 Calls, also ~15–25 Testläufe pro Tag. Tipp: gpt-4o-mini wählen – meist höheres Limit und für den Calculator-Agenten mehr als schlau genug. Für echte Agenten mit langen Loops wären 50/Tag zu wenig; dafür gibt es kostenpflichtige Nutzung mit Budget. Token wie ein Passwort behandeln – nicht ins Repo committen.

Alternative: dein eigenes Modell über dessen API einbinden

Du hast bereits einen API-Key bei OpenAI, Anthropic oder Google? Dann bind dein Modell direkt ein – Flowise hat dafür fertige Knoten: ChatOpenAI, ChatAnthropic, ChatGoogleGenerativeAI. Passenden Knoten wählen, deinen Key als Credential eintragen, Modell setzen (z. B. gpt-4o-mini, ein Claude-haiku, ein Gemini-flash) – fertig. Eine Basis-URL musst du dann nicht überschreiben.

Achtung – Abo ≠ API-Key: Ein Chat-Abo (ChatGPT Plus, Claude Pro, Gemini Advanced) enthält keinen API-Zugang. Den API-Key holst du dir separat im Entwickler-Portal des Anbieters (platform.openai.com, console.anthropic.com bzw. Google AI Studio); dort wird pro Nutzung abgerechnet – meist nur Cent-Beträge. Gratis-Tipp: Google AI Studio bietet für die Gemini-API einen echten kostenlosen Tier mit großzügigeren Limits als GitHub Models – eine gute Alternative, wenn dir 50/Tag zu knapp sind.

Erwartetes Ergebnis: Die Credential erscheint gespeichert in der Liste; sie lässt sich später im Knoten auswählen.

Schritt 2 · Einen Agentflow anlegen

Neuen Flow erstellen und eine Agent-Vorlage wählen (nicht einen reinen „Conversation Chain"-Chatflow). Kern-Knoten ist ein Tool Agent / AgentExecutor, dem du dein Chatmodell aus Schritt 1 zuweist. → Das ist Schicht 2: der Loop.

Erwartetes Ergebnis: Im Canvas liegt ein Agent-Knoten, der ein „Chat Model" als Eingang erwartet.

Schritt 3 · Das Calculator-Tool verbinden

Zieh den Knoten Calculator in den Canvas und verbinde ihn mit dem Tool-Eingang des Agenten. Sonst nichts – keine Websuche, keine Vektor-DB. → Ein Tool aus dem Harness (Modul 4), bewusst minimal.

Erwartetes Ergebnis: Der Agent listet „Calculator" als verfügbares Tool.

Schritt 4 · System-Prompt setzen

Gib dem Agenten ein kurzes Ziel, z. B.: Du bist ein hilfreicher Assistent. Nutze das Calculator-Tool für jede Rechnung, statt selbst zu rechnen. Flow speichern.

Schritt 5 · Testen – das Tool erzwingen

Chat öffnen und fragen: Was ist 1348 × 27 + 99?

Erwartetes Ergebnis: Die Antwort lautet 36 495. Wichtiger als die Zahl: im Verlauf/Log siehst du, dass der Agent das Calculator-Tool aufgerufen hat (mit dem Ausdruck als Parameter) und erst danach geantwortet hat.

Schritt 6 · Den Loop sichtbar machen

Öffne die Detailansicht des letzten Laufs (in Flowise meist „View Messages" / das Log-/Trace-Panel). Du erkennst die Sequenz:

run-log (vereinfacht)
# Reason  Agent: „Das ist eine Rechnung -> ich nutze Calculator."
# Act     tool: Calculator("1348 * 27 + 99")
# Observe result: 36495
# Final   Agent: „1348 × 27 + 99 = 36495."

Das ist der Reason-Act-Observe-Loop aus Modul 2 – live, an deinem eigenen Agenten.

Troubleshooting

„Invalid API key" / 401: Credential aus Schritt 1 nochmals prüfen; richtigen Anbieter im Modell-Knoten gewählt? Key ohne Leerzeichen eingefügt?

Agent antwortet, ohne das Tool zu nutzen: System-Prompt schärfen („nutze immer das Calculator-Tool für Rechnungen"); prüfen, ob das Tool wirklich mit dem Agenten verbunden ist (Linie im Canvas).

npx flowise start hängt oder bricht ab: Node-Version prüfen (node -v ≥ 18); Firmen-Firewall/Proxy kann npx blockieren → dann den Cloud-Tier nehmen.

Port 3000 belegt: npx flowise start --PORT=3100 und http://localhost:3100 öffnen.

Kein API-Key verfügbar: Viele Anbieter geben Gratis-Startguthaben; sonst Kollegen fragen oder den Cloud-Tier mit dortigem Free-Kontingent nutzen.

Abschlussaufgabe (das eigentliche Lernen)

Experiment: Tool-Beschreibung verschlechtern

Ändere die Beschreibung/den Namen des Calculator-Tools in etwas Nichtssagendes (z. B. „tool_x", Beschreibung leeren). Stelle dieselbe Rechenfrage erneut. Beobachtung: Der Agent nutzt das Tool nun seltener, falsch – oder rechnet selbst (und liegt evtl. daneben). Damit hast du am eigenen Agenten erlebt, warum Tool-Beschreibungen Teil des Harness-Engineerings sind (Modul 4).

Lab 1 geschafft, wenn …

  • dein Agent das Calculator-Tool von selbst aufgerufen hat
  • du im Log die Sequenz denken → Tool → Ergebnis → Antwort erkannt hast
  • du gesehen hast, wie eine schlechtere Tool-Beschreibung das Verhalten verändert

Lab 2 Spring-AI-Mini-Agent optional · Java-Transfer

Für die, die den Bezug zu unserem Stack wollen. Ziel: ein winziger Spring-Boot-Agent mit einem Tool (getOrderStatus) und Kurzzeitgedächtnis – lokal startbar, eine Frage, ein sichtbarer Tool-Call im Log.

Voraussetzungen & Version

JDK 17+, Maven, ein LLM-API-Key als Umgebungsvariable. Getestet gegen Spring AI 2.0.0 und Spring Boot 3.4.x. Da sich die API bewegt: Bei Abweichungen die Reference-Docs deiner Version konsultieren.

1 · Abhängigkeiten (pom.xml, Auszug)

pom.xmlSpring AI 2.0.0
<dependency>
    <groupId>org.springframework.ai</groupId>
    <artifactId>spring-ai-starter-model-openai</artifactId>
</dependency>
<!-- BOM verwaltet die Version: spring-ai-bom 2.0.0 -->

2 · Konfiguration (application.properties)

application.properties
# Variante A: OpenAI direkt
spring.ai.openai.api-key=${OPENAI_API_KEY}
spring.ai.openai.chat.options.model=gpt-4o-mini

# Variante B: kostenlos via GitHub Models (OpenAI-kompatibel)
# spring.ai.openai.api-key=${GITHUB_TOKEN}          # PAT mit Scope models:read
# spring.ai.openai.base-url=https://models.github.ai
# spring.ai.openai.chat.completions-path=/inference/chat/completions
# spring.ai.openai.chat.options.model=openai/gpt-4o-mini
GitHub Models auch hier nutzbar

Da GitHub Models OpenAI-kompatibel ist, genügt im Spring-AI-OpenAI-Starter ein Überschreiben von base-url, completions-path und Modell-ID (Variante B oben) – als „Key" der GitHub-PAT mit models:read. Exakte URL/Path/Modell-ID bitte gegen die aktuelle GitHub-Models-Doku und deine Spring-AI-Version prüfen.

3 · Tool + Agent + eine Frage

MiniAgent.javaSpring AI 2.0.0
@SpringBootApplication
public class MiniAgent {
  public static void main(String[] a){ SpringApplication.run(MiniAgent.class, a); }

  // --- das eine Tool ---
  static class OrderTools {
    @Tool(description = "Liefert den Versandstatus einer Bestellung per ID.")
    String getOrderStatus(String orderId) {
      return "4711".equals(orderId) ? "versandt am 24.06." : "unbekannt";
    }
  }

  // --- Agent: ChatClient mit Memory; Tools werden in 2.0 automatisch geloopt ---
  @Bean
  CommandLineRunner run(ChatClient.Builder builder) {
    ChatClient agent = builder
        .defaultAdvisors(MessageChatMemoryAdvisor
            .builder(MessageWindowChatMemory.builder().build()).build())
        .defaultTools(new OrderTools())
        .build();

    return args -> {
      String answer = agent.prompt()
          .user("Wie ist der Status von Bestellung 4711?")
          .call().content();
      System.out.println(answer);
    };
  }
}

4 · Starten & den Tool-Call im Log sehen

Terminal
$ export OPENAI_API_KEY=sk-...
$ ./mvnw spring-boot:run
...
DEBUG o.s.a.tool  : calling tool 'getOrderStatus' args={orderId=4711}
DEBUG o.s.a.tool  : tool result: versandt am 24.06.
> Die Bestellung 4711 wurde am 24.06. versandt.

Tipp: logging.level.org.springframework.ai=DEBUG setzen, um Tool-Calls im Log zu sehen – euer erster, einfachster „Trace" (mehr in Modul 8).

Kleine Transferaufgabe

Verbessere die @Tool-Beschreibung (z. B. präziser, mit Hinweis aufs ID-Format) und stelle eine knifflige Frage („Status der Bestellung mit Nummer 4711?"). Beobachte im DEBUG-Log, ob das Tool zuverlässiger getroffen wird. Gleiche Lektion wie in Lab 1 – diesmal in Java.

Merksatz
Der Moment, in dem du im Log siehst, wie der Agent selbst entscheidet ein Tool zu rufen, ist der Moment, in dem „Agentic AI" vom Buzzword zur greifbaren Sache wird – ob im No-Code-Canvas oder in Spring.
Bitte beachten

Nutze für beide Labs keine echten Firmen-/Kundendaten und keine produktiven API-Keys. Ein Wegwerf-Key mit Gratis-Guthaben und Spielzeug-Eingaben genügt. Warum das wichtig ist, klärt Modul 11 (Security).
