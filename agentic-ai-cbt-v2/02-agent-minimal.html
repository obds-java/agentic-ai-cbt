2 Definition

Was macht einen Agenten zum Agenten?

„Agent" ist das meistmissbrauchte Wort der Branche. Hier die Minimalbedingungen – ab wann der Begriff wirklich gerechtfertigt ist.

Nach diesem Modul kannst du …

  • die vier Minimalbestandteile eines Agenten aufzählen
  • den Agent-Loop (Reason → Act → Observe) erklären
  • das ReAct-Muster einordnen
  • einen „Pseudo-Agenten" (nur Prompt-Chaining) von einem echten unterscheiden

2.1 Die Definition von „agentisch"

„Agentisch" beschreibt jedes System, in dem ein LLM Agency zeigt: Es plant, entscheidet, nutzt Tools und passt sich auf ein Ziel hin an – ohne dass ein Mensch jeden Schritt vorgibt. Das ist weniger ein Ja/Nein als ein Grad. Die spannende Frage ist deshalb: Was muss minimal vorhanden sein, damit man von einem Agenten sprechen darf?

2.2 Die vier Minimalbestandteile

Streiche einen davon, und du hast keinen Agenten mehr, sondern einen Chatbot oder ein Skript:

🎯

1 · Ein Ziel

Ein Zielzustand, der mehr als einen Schritt erfordert. „Beantworte X" ist kein Agentenziel – „bring die Reise von A nach B unter 300 € zustande" schon.

🧠

2 · Das Modell als Entscheider

Das LLM wählt den nächsten Schritt selbst, statt ihn vom Code diktiert zu bekommen. Hier sitzt die Autonomie.

🔧

3 · Aktionen / Tools

Der Agent kann auf die Welt einwirken: API aufrufen, Datei schreiben, Suche absetzen. Ohne Aktionen bleibt „Denken" folgenlos.

🔁

4 · Die Schleife (Feedback)

Der Agent beobachtet das Ergebnis seiner Aktion und entscheidet auf dieser Basis neu – iterativ, bis das Ziel erreicht oder ein Limit erreicht ist.

Faustregel

Wenn du den Ablauf als festes Flussdiagramm aufzeichnen kannst, das immer gleich läuft, ist es kein Agent – egal wie viele LLM-Calls drinstecken. Ein Agent ist erkennbar daran, dass die Anzahl und Reihenfolge der Schritte vorab nicht feststeht.

2.3 Der Agent-Loop

Das Herzstück jedes Agenten ist eine Schleife. Klassisch: Reason → Act → Observe (oft „ReAct" genannt). Das Modell denkt nach, führt eine Aktion aus, sieht das Resultat, denkt erneut – so lange, bis es entscheidet, dass es fertig ist.

In der Praxis prüft der Loop nach jeder Runde eine Stop-Bedingung: Ziel erreicht, Schrittlimit überschritten, Budget aufgebraucht oder Fehler. Ohne saubere Stop-Bedingung kann ein Agent in Endlosschleifen laufen – ein häufiger und teurer Anfängerfehler (mehr dazu in Modul 10).

agent-loop.txt
while ziel_nicht_erreicht and schritte < LIMIT:
    gedanke, aktion = llm(ziel, verlauf, verfügbare_tools)   # Reason
    if aktion == "fertig":
        break
    ergebnis = tool_ausführen(aktion)                       # Act
    verlauf.append(gedanke, aktion, ergebnis)              # Observe
    schritte += 1
Begriff: ReAct

ReAct = „Reasoning + Acting". Das Modell verschränkt im selben Kontext explizite Gedanken („Ich sollte zuerst den Lagerbestand prüfen") mit Tool-Aufrufen. Dieses Muster ist bis heute das mentale Grundmodell für die meisten Agenten – auch wenn moderne Frameworks es unterschiedlich umsetzen. (Hinweis: nicht zu verwechseln mit dem JS-Framework React.)

2.4 Echter Agent vs. Pseudo-Agent

FragePseudo-Agent (Chain)Echter Agent
Steht die Schrittfolge vorab fest?janein
Wählt das Modell die Tools selbst?nein, Code wähltja
Reagiert es auf unerwartete Zwischenergebnisse?nur wie programmiertja, plant neu
Kann die Schrittzahl variieren?neinja
Merksatz
Ziel + eigenständige Entscheidung + Aktionen + Feedback-Schleife. Fehlt eines davon, ist es kein Agent – sondern ein Chatbot oder ein Skript mit LLM-Würze.

Richtig: A. Ohne Stop-Bedingung (Ziel/Schrittlimit/Budget/Fehler) schleift ein Agent endlos. Genau dafür braucht jeder produktive Loop harte Limits.

Richtig: B & C. Keine Limits = Endlosschleifen-Risiko; verschluckte Fehler = der Agent kann nicht umplanen. A und D sind genau richtig.
